Issue 1: Thay delay hardcode bằng Wait Element

Hiện trạng

Recorder tự gắn delay: 1000 cho mọi step (click, fill, selectOption, keyboard) trong recorder.ts dòng 1546–1610. Quan trọng: runner đã wait element rồi — click chờ waitFor visible, fill click trước khi fill, selectOption chờ attached. Vậy delay: 1000 chỉ là padding THỪA chồng lên cơ chế wait sẵn có → chậm vô ích, không giải quyết flaky.

Giải pháp (đơn giản hóa sau review)

  • Bỏ delay: 1000 mặc định trong actionToStep() (4 chỗ: click, fill, selectOption, keyboard). Đây là fix chính.
  • Dựa vào wait sẵn có của runner — không cần thêm waitForDomStable() như bản plan đầu (runner đã chờ element visible/attached trước mỗi action).
  • Bổ sung wait cho action thiếu: keyboard hiện không wait gì. Thêm: nếu step trước là điều hướng → chờ networkidle nhẹ. Còn lại giữ nguyên.
  • Giữ delay field trong schema cho edge case (animation chậm) nhưng recorder KHÔNG tự thêm. Flow cũ có delay vẫn chạy (backward compat).

File changes

src/recorder.ts   — actionToStep(): xóa `delay: 1000` ở 4 case (click/fill/selectOption/keyboard)
src/runner.ts     — (tùy chọn) keyboard case: chờ load state nếu cần
src/types.ts      — không đổi (delay vẫn giữ cho manual override)

Exit criteria

  • Recorder không sinh delay trong YAML output trừ khi user explicit chỉ định.
  • Runner chạy flow cũ (có delay) vẫn pass backward-compatible.
  • Flow mới (không delay) chạy nhanh hơn ≥30% so với flow cũ cùng nội dung, không tăng tỉ lệ fail.

Issue 2: Auto-dismiss Popup bất ngờ

Hiện trạng

Runner chỉ dismiss browser dialog (alert/confirm/prompt) và OneSignal notification. Khi web app hiển thị modal/popup tự custom (vd: quảng cáo, survey, cookie consent, toast overlay) → tester click vào vùng trống bên ngoài popup để tắt → recorder ghi nhận click vào coordinate "trống" → khi replay, popup chưa kịp hiện hoặc đã đóng → step fail.

Giải pháp

  1. Runner — Popup Interceptor: inject MutationObserver theo dõi element mới xuất hiện match pattern popup phổ biến: 
    // Selectors phổ biến cho popup/modal/overlay
const POPUP_PATTERNS = [
  '.ant-modal-mask', '.ant-modal-wrap',
  '.modal-backdrop', '.modal.show',
  '[class*="popup"]', '[class*="overlay"]',
  '[class*="cookie"]', '[class*="consent"]',
  '[role="dialog"]', '[role="alertdialog"]',
  '.ant-notification', '.ant-message',
];
  2. Khi phát hiện popup → auto-dismiss:
    • Tìm nút close (✕, "Đóng", "Close", "Cancel", "Bỏ qua") → click.
    • Nếu không có nút close → click vào mask/backdrop (nếu clickable để dismiss).
    • Nếu popup là notification/toast → chờ tự mất (timeout 5s).
  3. Recorder side: khi user click vào mask/overlay → KHÔNG ghi step "click ở vị trí trống". Detect click target là backdrop → skip recording action đó, thay bằng ghi log [auto-dismiss] popup detected.
  4. Config: cho phép disable auto-dismiss qua config.autoPopupDismiss: false nếu flow cần test popup behavior.

File changes

src/runner.ts     — thêm injectPopupDismisser() vào page context khi launch
src/recorder.ts   — detect click on overlay → skip/mark as popup-dismiss
src/types.ts      — thêm autoPopupDismiss? vào FlowConfig

Exit criteria

  • Runner tự dismiss popup Ant Modal khi xuất hiện giữa flow → step tiếp theo không bị block.
  • Runner tự dismiss toast/notification → không ảnh hưởng click element bên dưới.
  • Recorder không ghi click "vào vùng trống" khi tắt popup → YAML sạch hơn.

Issue 3: Tối ưu bắt Element theo text động

Hiện trạng — PHẦN LỚN ĐÃ CÓ ✅

Sau khi đọc recorder.ts, hệ thống xử lý text động đã được implement khá đầy đủ:

  • isVolatileText() (dòng 774) — bắt "tồn", "số lượng/SL", giá đ/₫, số + đơn vị (viên/vỉ/hộp/ống...), chuỗi số ≥3 chữ số.
  • classifyAnchor() (dòng 660) — phân biệt anchor STATIC (label/header) vs DATA (cell dữ liệu).
  • Auto-variabilize (dòng 829) — anchor DATA động → tự đẩy thành {{var}}, tên biến lấy từ table header (deriveVarName) hoặc form label, push vào flow.variables.
  • VERIFY→AUTO-PICK lazy tiering — sinh locator theo bậc độ bền (flat → relative → css), rankFlat/anchorRank phạt volatile text → tự né bắt theo số tiền/tồn kho.

→ Plan gốc ("thêm isDynamicText, ưu tiên testId>role>relative") trùng với cái đã có. Scope thật sự chỉ còn vá vài gap nhỏ.

Gap còn lại (scope thu hẹp)

  1. [Gap chính] Assert mode ghi giá trị động literal. Khi user dùng Assert mode (Ctrl+Shift+A) click vào số tiền/tồn kho → recorder ghi expected: "1,234,000đ" nguyên văn (recorder.ts ~1140). Replay với data khác → fail.
    • Fix: trong assert handler, chạy isVolatileText(expectedValue) → nếu volatile, cảnh báo + đề xuất parameterize thành {{var}}, hoặc đổi assertType: 'text' sang 'visible' (chỉ verify element tồn tại, không so giá trị).
  2. [Gap nhỏ] Mở rộng isVolatileText patterns. Hiện sót: % đứng riêng, $/USD/ngoại tệ, ngày dd/mm/yyyy đứng độc lập (dù năm 4 số bị \d{3,} bắt một phần).
    • Fix: thêm pattern $/USD/€, \d+%, ngày \d{1,2}[/\-.]\d{1,2}[/\-.]\d{2,4}.

Issue 4: Hover → hiện nút bấm → bắt action

Hiện trạng

Khi user hover vào icon/row → các action buttons mới hiện ra (visibility: visible / display: block) → user click button đó. Hiện tại recorder không ghi nhận hover event (chỉ ghi click/fill/keyboard). Runner hỗ trợ action: 'hover' nhưng recorder không emit nó → khi replay, button chưa hiện → click fail.

Giải pháp

  1. Recorder — detect hover-to-reveal pattern:
    • Khi user click vào element mà trước click 500ms, element đó chưa visible (hoặc cha/ông nó mới visible) → suy ra đã có hover trigger.
    • Inject script track mouseenter event trên mọi element → lưu last-hovered element.
    • Khi click xảy ra trên element vừa mới visible → tự chèn step hover trước step click trong actions list.
  2. Heuristic detect hover target: element được hover = tổ tiên gần nhất của click target mà trigger visibility change. Tìm bằng cách:
    • Track mouseenter → ghi lại ancestor chain.
    • Khi visibility change detected (MutationObserver on style/class) → map lại ancestor nào trigger sự thay đổi.
  3. Runner — hover + waitFor: step hover đã hỗ trợ. Thêm implicit wait sau hover: chờ child element visible trước khi proceed sang step tiếp.

File changes

src/recorder.ts   — inject script: track mouseover (debounced), detect hover-reveal, emit hover action
src/recorder.ts   — buildFlow(): insert hover step trước click khi detect pattern
src/runner.ts     — hover case: thêm wait cho revealed element sau hover
⚠️ Phần khó nhất trong 5 issue. Detect "click target vừa được reveal bởi hover" dễ false-positive (mouseover fire liên tục). Đề xuất tiếp cận an toàn:
  • Track mouseover với debounce, lưu element hover gần nhất + thời điểm.
  • Trong pointerup handler: nếu click target (hoặc ancestor gần) không visible ở thời điểm 1 hover trước đó và chỉ visible sau khi hover ancestor → mới emit hover step.
  • Nếu không chắc → KHÔNG emit (thà thiếu hover step còn hơn ghi hover thừa làm flow rối). Có thể bổ sung phím tắt thủ công "ghi hover" nếu auto-detect chưa đủ tin cậy.
Nếu auto-detect quá flaky trong thực nghiệm → fallback sang phương án thủ công (hotkey đánh dấu hover) thay vì cố ép heuristic.

Exit criteria

  • User hover icon → button hiện → click button → recorder ghi: step hover + step click (2 steps).
  • Runner replay: hover → chờ button visible → click → pass.
  • Không false positive: hover bình thường (không reveal gì) → KHÔNG ghi hover step thừa.

Issue 5: Cross-platform CLI (Windows support)

Hiện trạng

Command frt-test wire qua npm link (field bin trong package.json). Nhưng bin/frt-test là bash script (#!/usr/bin/env bash). Trên Windows, npm sinh shim .cmd bọc script này nhưng khi chạy lại cố gọi qua bash — không có Git Bash/WSL trên PATH → fail → agent buộc phải tạo frt-test-win riêng. README dòng 50 cũng chỉ hướng dẫn npm link cho macOS/Linux.

Giải pháp (sạch hơn plan đầu)

Thay vì maintain 2 file (frt-test bash + frt-test.cmd thủ công, dễ desync), đổi bin/frt-test thành Node.js launcher. Khi bin là Node script, npm link/npm install tự sinh shim frt-test (unix) + frt-test.cmd + frt-test.ps1 (Windows) → cross-platform tự động, 1 nguồn duy nhất.

#!/usr/bin/env node
// bin/frt-test — cross-platform launcher
const { spawnSync } = require('node:child_process');
const path = require('node:path');
const engineDir = path.resolve(__dirname, '..');
const tsxBin = path.join(engineDir, 'node_modules', 'tsx', 'dist', 'cli.mjs');
const r = spawnSync(process.execPath, [tsxBin, path.join(engineDir, 'src', 'cli.ts'), ...process.argv.slice(2)],
  { stdio: 'inherit' });
process.exit(r.status ?? 1);
  • Gọi tsx/dist/cli.mjs trực tiếp bằng process.execPath (node) → không phụ thuộc .bin/tsx symlink (vốn cũng là bash trên unix, .cmd trên win).
  • package.json bin giữ nguyên "frt-test": "./bin/frt-test" — npm tự lo phần shim.
  • Bỏ logic resolve symlink BSD trong bash script cũ (không cần — Node __dirname đã chuẩn cross-platform).

File changes

docs/guide-vi.html — cập nhật bước setup Windows

Exit criteria

  • Trên Windows (cmd.exe / PowerShell): sau npm link, frt-test --help chạy được không cần WSL/Git Bash, không cần file frt-test-win.
  • Trên macOS/Linux: frt-test vẫn chạy như cũ (verify frt-test list, frt-test play).
  • Không còn file/script đặc thù Windows nào cần maintain riêng.

Risks & Notes

  • Backward compatibility: flow YAML cũ có delay: 1000 phải vẫn chạy được. Runner giữ support delay field, chỉ recorder không tự sinh.
  • Popup false positive: auto-dismiss có thể đóng popup quan trọng (confirm delete…). Giải pháp: chỉ dismiss popup "không có action button nguy hiểm" (không có "Xóa"/"Delete"). Config autoPopupDismiss: false cho flow cần test popup.
  • Hover detection accuracy: mouseenter fire liên tục khi di chuột. Cần debounce + chỉ emit khi CÓ visibility change thực sự xảy ra.
  • Windows path separator: các hàm dùng path.join / path.resolve — Node.js tự handle. Chỉ cần check nơi nào dùng hardcode /.
