1. Context & Vấn đề

Hiện trạng

Template frt-test-recorder-template hiện tại được onboard cho mỗi chuỗi bằng lệnh cp -r frt-test-recorder-template frt-test-recorder-{chain}. Tức là mỗi chuỗi có 1 bản sao đầy đủ của engine code (src/cli.ts, src/runner.ts, src/recorder.ts, …).

Pain point

  • Khi engine có bug fix / feature mới (vd: thêm action type, fix selector resolver, support browser khác) → phải đi update N repo riêng biệt.
  • Drift, version mismatch giữa các chuỗi.
  • Không có single source of truth cho engine logic.

Lợi thế hiện có

Engine code đã được viết theo pattern "thin CLI" — tất cả paths đều resolve qua process.cwd():

  • cli.ts: path.join(process.cwd(), 'flows'), path.join(process.cwd(), 'reports')
  • config.ts: path.join(process.cwd(), 'config/environments.yaml')
  • runner.ts, recorder.ts: cùng pattern
Hệ quả: engine không phụ thuộc vào việc nó nằm ở đâu, chỉ cần được chạy từ folder của chuỗi (chứa flows/, config/, screenshots/, reports/). Không cần refactor logic — chỉ cần tách packaging.

2. Mục tiêu

  1. R1 — Engine code maintain ở 1 nơi duy nhất (ai-test/recorder-engine).
  2. R2 — Chain repos chỉ chứa data (config/, flows/), KHÔNG có engine code, KHÔNG có node_modules/.
  3. R3 — Tester workflow đơn giản: clone engine + chain → npm link → chạy frt-test.
  4. R4 — Onboarding chuỗi mới hoàn toàn tự động qua command (Node.js, cross-platform).
  5. R5 — Update engine = git pull. Không cần npm publish, không cần version pin.
  6. R6 — Backward compat: command surface giữ nguyên (record, play, list, convert với same flags).
  7. R7 — Cross-platform: macOS / Linux primary. Windows qua Git Bash hoặc WSL.

3. Decisions đã chốt

IDDecisionRationale
D1 Distribution = git clone + npm link (KHÔNG dùng npm package + version pin) Đơn giản hơn cho team nội bộ, không cần npm registry, debug-friendly (tester có local source code).
D2 Engine ships raw TS + tsx (không build dist/) Tester có local source. Update = git pull. Không cần build pipeline. Dev experience giữ nguyên.
D3 Onboarding script là CLI subcommand trong engine: frt-test scaffold-chain Cross-platform free (Node.js), reuse commander infrastructure, không cần shell variant cho Windows.
D4 LAB chain — làm mới từ đầu, không có migration Repo cũ frt-test-recorder-lab chưa có flows. User confirmed.
D5 Versioning: chain repos không pin engine version Tester git pull engine để update. Engine in version banner mỗi lần chạy + warn nếu commit cũ > 14 ngày.
D6 CI/CD regression: defer Phase 2 Phase 1 manual test trước. CI thêm sau khi pattern stabilize.
D7 Chain template lives inside engine repo (chain-template/ subfolder) Collocated với scaffold command + version cùng engine. Khi engine bump pattern → chain-template bump theo.
D8 Documentation chính = prompt cookbook cho AI agent (không phải technical CLI manual) Tester là user cuối, không phải dev. Họ paste prompt vào AI agent (Kiro / Cursor / Claude / Codex), agent gọi shell + git + npm + frt-test. Match AI-native SDLC pattern. Tester không cần học CLI flags.

5. Workflow của tester (qua AI agent)

Nguyên tắc: Tester KHÔNG gõ shell commands trực tiếp. Tester paste prompt tiếng Việt vào AI agent (Kiro / Cursor / Claude / Codex), agent tự gọi tools (shell, file edit, git) để execute.

Tất cả prompts được document trong engine repo tại docs/AGENT_PROMPTS.md và docs/prompts/. Tester chỉ cần copy prompt phù hợp với task, điền variables, paste vào agent chat.

Setup 1 lần (mỗi máy tester)

Prompt cho agent: 
Setup engine recorder cho tôi.
Username GitLab của tôi: {your-gitlab-username}
Workspace folder: ~/work

Agent sẽ tự thực hiện:

  1. Verify Node.js >= 18, Git, npm có sẵn (báo nếu thiếu)
  2. git clone git@git2.fptshop.com.vn:ai-test/recorder-engine.git ~/work/recorder-engine
  3. cd ~/work/recorder-engine && npm install
  4. npx playwright install chromium
  5. npm link
  6. Verify frt-test --version in version banner
  7. Báo done + lệnh tiếp theo

Daily work — Record test mới

Prompt: 
Record test mới cho chuỗi LAB, app rsa-lab, tên: "Tạo đơn hàng dịch vụ XYZ"

Agent sẽ:

  1. Verify cwd hoặc cd ~/work/frt-test-lab (clone về nếu chưa có)
  2. Verify rsa-lab có trong config/environments.yaml
  3. Chạy frt-test record --app rsa-lab -n "Tạo đơn hàng dịch vụ XYZ"
  4. Hướng dẫn tester thao tác (chậm rõ ràng, đợi 1-2s sau dropdown, Ctrl+Shift+A để add assertion)
  5. Sau Ctrl+C, verify file YAML tạo thành công

}

6.2 — bin/frt-test (MỚI)

#!/usr/bin/env bash
# Shell wrapper để npm link expose command `frt-test` toàn cục.
# Resolve symlink để tìm engine root (BSD-compatible cho macOS).

SOURCE="${BASH_SOURCE[0]}"
while [ -h "$SOURCE" ]; do
  DIR="$( cd -P "$( dirname "$SOURCE" )" && pwd )"
  SOURCE="$(readlink "$SOURCE")"
  [[ $SOURCE != /* ]] && SOURCE="$DIR/$SOURCE"
done
ENGINE_DIR="$( cd -P "$( dirname "$SOURCE" )/.." && pwd )"

exec node --import tsx "$ENGINE_DIR/src/cli.ts" "$@"

Cần chmod +x bin/frt-test. Windows users dùng Git Bash để chạy script này, hoặc npm tự generate frt-test.cmd shim trong %APPDATA%\npm.

9. Scaffold command (onboarding tự động)

Usage

frt-test scaffold-chain \
  --name lab \
  --display "LAB (Xét nghiệm)" \
  --apps "rsa-lab=https://ci-lab-rsa.frt.vn,ecom-lab=https://ci-lab-ecom.frt.vn,portal-lab=https://ci-lab-portal.frt.vn" \
  --output ~/work/frt-test-lab \
  --remote git@git2.fptshop.com.vn:ai-test/frt-test-lab.git \
  --push

Flow của command

  1. Validate args (chain name kebab-case, apps format name=url, output dir không tồn tại hoặc empty).
  2. mkdir -p ~/work/frt-test-lab.
  3. Copy chain-template/* sang output dir.
  4. Render templates (.tpl → file thật) bằng simple {{VAR}} substitution:
    • {{CHAIN_NAME}} → "LAB (Xét nghiệm)"
    • {{CHAIN_LOWER}} → "lab"
    • {{APPS_YAML}} → multi-line YAML block của apps
    • {{APPS_TABLE}} → markdown table cho README
    • {{DEFAULT_APP}} → app đầu tiên
  5. mkdir -p flows/<app>/_shared cho mỗi app.
  6. git init + git add . && git commit -m "init: scaffold {chain} chain".
  7. Nếu có --remote: git remote add origin <url>.
  8. Nếu có --push: git push -u origin main.
  9. In summary: path, apps đã setup, lệnh tiếp theo cho tester.

Tham khảo: thêm app vào chuỗi đã có

cd ~/work/frt-test-lab
frt-test add-app --name admin-lab --url https://admin-ci-lab.frt.vn
# → Thêm entry vào config/environments.yaml
# → Tạo flows/admin-lab/_shared/
# → Echo lệnh git tiếp theo

Source code mới: src/scaffold.ts

Functions cần có:

  • scaffoldChain(opts: ScaffoldOpts): Promise<void>
  • addAppToChain(opts: AddAppOpts): Promise<void>
  • renderTemplate(content: string, vars: Record<string,string>): string — simple {{VAR}} replacement, no template engine dep.
  • copyTemplateDir(src: string, dst: string, vars): Promise<void>
  • execGit(cwd: string, args: string[]): Promise<string> — wrapper child_process.execFile.

Wire vào src/cli.ts như program.command('scaffold-chain') với commander, theo pattern của record/play.

11. Verification & exit criteria

Manual smoke test cuối Phase 4 — qua AI agent

Một tester (KHÔNG phải author) dùng AI agent (Kiro / Cursor / Claude / Codex) để execute toàn bộ flow:

  1. Mở agent, paste prompt #01 ("Setup engine recorder cho tôi. Username GitLab: ...") → agent clone + install + npm link → verify frt-test --version OK.
  2. Paste prompt #02 (admin paste, "Onboard chuỗi LAB ... apps: ...") → agent gọi scaffold + push lên ai-test/frt-test-lab.
  3. Paste prompt #04 ("Record test mới cho LAB, app rsa-lab, tên: Login test") → agent mở browser → tester thao tác → file YAML tạo thành công.
  4. Paste prompt #05 ("Play test login-test của LAB") → test chạy pass.
  5. Paste prompt #08 ("Commit và push test login-test lên repo LAB") → flows/ commit + push thành công.
  6. Test fail intentional: tester sửa selector sai trong YAML → paste prompt #06 ("Test login-test fail, fix giúp tôi") → agent diagnose + fix → re-run pass.
Tiêu chí pass: Tester KHÔNG được gõ shell command trực tiếp trong toàn bộ smoke test. Mọi action đi qua prompt. Nếu agent stuck → root cause là prompt file thiếu thông tin → bổ sung vào prompt file đó.

Definition of Done (toàn project)

  • ✓ 9 repos visible trong ai-test/ group, đúng structure.
  • ✓ Engine code chỉ tồn tại ở ai-test/recorder-engine (không có src/ trong chain repos).
  • ✓ 10 prompt files trong docs/prompts/ + README quick start.
  • ✓ Tester KHÔNG cần gõ shell command — mọi task qua prompt cho AI agent.
  • ✓ Tester độc lập setup + record + play + commit qua prompt < 20 phút.
  • ✓ Update engine = paste prompt #09, agent tự git pull.
  • ✓ Onboard chuỗi mới = paste prompt #02 với 3 dòng app info.
  • ✓ Folder frt-test-recorder-template/ ở repo workspace đã được xử lý.

12. Risks & mitigations

RiskSeverityMitigation
npm link issues trên Windows (PATH, permissions) Medium Document Git Bash + WSL fallback. Test trên Windows machine trước Phase 4.
Tester forget git pull engine → dùng version cũ → behavior khác bạn QC Medium Engine in version banner mỗi run + warn nếu commit cũ > 14 ngày (D5).
Multiple engine versions cần thiết cùng lúc (vd: chuỗi A cần engine v1, chuỗi B cần v2) Low (chưa xảy ra) Single global symlink — limitation chấp nhận. Nếu xảy ra: tester clone 2 folder engine khác nhau, dùng path tuyệt đối thay vì npm link.
Engine breaking change (vd: đổi YAML schema) High Engine bump major version + maintain backward-compat parser cho 1 major version. CHANGELOG.md bắt buộc.
Scaffold command ghi đè files có sẵn Medium Validate output dir empty trước scaffold. add-app command refuse nếu app đã exist.
Tester không có access push lên ai-test group Low Coordinate với IT để add tester vào group permissions trước Phase 4.
Prompt file thiếu detail → agent execute sai / stuck High Smoke test Phase 4 bằng tester KHÔNG phải author + agent KHÔNG phải agent của author. Mỗi failure → bổ sung "Common errors & fix" vào prompt file. Iterate đến khi 1 tester làm trôi chảy.
AI agent khác nhau (Kiro/Cursor/Claude) execute prompt khác nhau Medium Prompt agent-agnostic — chỉ assume shell + file edit access. Test với ít nhất 2 agent khác nhau ở Phase 4 (vd: Kiro + Cursor).
