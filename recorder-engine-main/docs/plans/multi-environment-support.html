1. Context — vấn đề hiện tại

Engine recorder hiện tại đã lên 8 chuỗi (LAB, Pharmacy, ICT, DS, BO, Data, Platform, VAC). Mỗi chuỗi có nhiều app (RSA, ecom, portal…), nhưng schema config config/environments.yaml chỉ map mỗi app sang một URL duy nhất:

# Pharmacy hiện tại
apps:
  rsa:
    baseUrl: "https://ci-rsa-web.frt.vn"      # ⚠️ chỉ CI
  rsa-ecom:
    baseUrl: "https://ci-rsa-ecom.frt.vn"     # ⚠️ chỉ CI
default_app: rsa

Hệ quả:

  • Test YAML hardcode URL CI: config.baseUrl: https://ci-rsa-web.frt.vn + step goto value: https://ci-rsa-web.frt.vn.
  • Muốn chạy test trên UAT/PROD → phải sửa file YAML hoặc sửa environments.yaml bằng tay.
  • Không thể có pipeline regression: "chạy bộ test pharmacy/rsa trên UAT trước khi promote PROD" mà không hack tạm.
Tên gọi đang lẫn lộn: file đặt tên environments.yaml nhưng nội dung lại là apps (theo schema mới). Đây là di sản từ schema cũ (xem config.ts còn xử lý cả environments key lẫn apps key). Plan này sẽ giữ tên file để không phá backward-compat của 8 chains, nhưng nội dung sẽ thêm tầng urls.{env}.

2. Goals & Non-goals

Goals

  • Schema config hỗ trợ nhiều env per app (CI / UAT / PROD).
  • Tester chọn env qua CLI flag required: frt-test play --app rsa --env uat <file>. Không có default → ép tester gõ rõ → tránh nhầm chạy lên PROD.
  • Mỗi env có file YAML riêng trong folder riêng: flows/{app}/{env}/<name>.yaml. Record xong cho CI ≠ record xong cho UAT.
  • Recorder mở browser tới URL của (app, env), lưu YAML vào folder env tương ứng.
  • Scaffold chain mới hỗ trợ nhập nhiều env per app, env có thể "chưa có URL" (placeholder để bổ sung sau).
  • Backward-compat: 8 chains đã onboard chạy được sau update engine. Migration là opt-in.
  • Có command helper để user clone test CI sang UAT làm seed: clone-flow --from ci --to uat <file>. KHÔNG tự động assume test CI chạy được UAT.
  • Support env var FRT_TEST_ENV=uat cho CI/CD pipeline (ngoài CLI flag).

Non-goals

  • ❌ Không auto-parametrize URL trong YAML. Test CI và UAT là 2 file YAML độc lập, có thể có DOM/flow khác nhau (theo yêu cầu user).
  • ❌ Không có default env. Phải truyền --env mỗi lần.
  • ❌ Không tách env-specific data vào nhiều file config (vd environments.ci.yaml) — giữ 1 file environments.yaml.
  • ❌ Không thêm logic secrets per-env (credentials login UAT khác PROD) — plan riêng.
  • ❌ Không support per-step env override.
  • ❌ Không thêm dependency mới.

Resolve algorithm

function resolveBaseUrl(app: string, env: string): string {
  const appCfg = config.apps[app];
  if (!appCfg) throw new Error(`App "${app}" không có trong config`);

  // Schema v2 (urls map)
  if (appCfg.urls) {
    if (!(env in appCfg.urls)) {
      const available = Object.keys(appCfg.urls).join(', ');
      throw new Error(
        `App "${app}" không định nghĩa env "${env}". Có sẵn: ${available}`
      );
    }
    const url = appCfg.urls[env];
    if (url == null || url === '') {
      throw new Error(
        `App "${app}" / env "${env}": URL chưa được set (giá trị null). ` +
        `Sửa config/environments.yaml hoặc dùng: frt-test add-env --app ${app} --env ${env} --url `
      );
    }
    return url;
  }

  // Schema v1 (legacy: single baseUrl string) — chỉ accept env="default"
  if (appCfg.baseUrl) {
    if (env && env !== 'default') {
      throw new Error(
        `Chuỗi này dùng schema v1 (chưa migrate). --env "${env}" không support. ` +
        `Migrate: frt-test migrate-config`
      );
    }
    return appCfg.baseUrl;
  }

  throw new Error(`App "${app}" không có baseUrl/urls`);
}

function resolveEnv(envOpt?: string): string {
  // Priority: CLI flag > env var. KHÔNG có default.
  const env = envOpt || process.env.FRT_TEST_ENV;
  if (!env) {
    throw new Error(
      'Thiếu --env. Phải chỉ định môi trường: --env ci|uat|prod ' +
      '(hoặc set FRT_TEST_ENV)'
    );
  }
  return env;
}

5. Thay đổi theo file

src/config.ts

  • Mở rộng interface AppConfig: thêm urls?: Record<string, string> bên cạnh baseUrl?: string.
  • Mở rộng EnvConfig: thêm environments?: Record<string, EnvMeta>, default_env?: string, schema_version?: number.
  • Sửa resolveBaseUrl(app?, env?) theo algorithm ở §3. Hiện tại signature là resolveBaseUrl(app?) → đổi thành resolveBaseUrl(app?, env?).
  • Thêm helper listEnvironments(): string[] + resolveEnv(envOpt?): string.

src/cli.ts

  • Thêm option -E, --env <env> cho play + record commands. Required ở runtime (resolveEnv throw nếu thiếu).
  • Sửa resolveBaseUrl(opts.app) → resolveBaseUrl(opts.app, env) với env = resolveEnv(opts.env).
  • Cập nhật flowsDir: flows/{app}/{env}/ thay vì flows/{app}/. rootFlowsDir giữ nguyên (cho shared resolve).
  • Cập nhật meta trong runOptions: thêm env để reporter render.
  • Banner khi chạy: in rõ App: rsa | Env: uat | URL: https://....
  • Thêm command mới: migrate-config, envs, add-env, clone-flow.

src/runner.ts

  • Constructor: FlowRunner(flowsDir, rootFlowsDir) — flowsDir trỏ vào flows/{app}/{env}/.
  • Trong run(): nhận thêm options.meta.env. Inject baseUrl, env, app vào vars (để YAML có thể optional dùng {{baseUrl}} nếu muốn — không bắt buộc): 
    const vars: FlowVariable = {
  ...(flow.variables || {}),
  baseUrl: options?.meta?.baseUrl || flow.config?.baseUrl || '',
  env: options?.meta?.env || '',
  app: options?.meta?.app || '',
  ...(options?.variables || {}),
};
  • RunResult.env?: string — lưu env đã chạy vào kết quả để reporter hiển thị.
  • listFlows(): scan recursive trong flows/{app}/{env}/ như cũ.
  • runAll(env): chỉ chạy YAML trong folder env tương ứng.

src/types.ts

  • FlowConfig: baseUrl giữ nguyên (legacy + tùy chọn).
  • RunResult: thêm env?: string.
  • Thêm interface EnvironmentMeta { label?: string; description?: string }.
  • Thêm field optional vào FlowFile: recorded_against?: { app: string; env: string; url: string; date: string } (metadata).

src/recorder.ts

  • Constructor + start(): nhận thêm env. Resolve URL = resolveBaseUrl(app, env).
  • Save vào folder flows/{app}/{env}/<name>.yaml (không phải flows/{app}/ như cũ).
  • KHÔNG tự thay URL bằng {{baseUrl}}. Giữ nguyên URL hardcode trong step goto. (Theo Q3 của user: mỗi env = 1 record riêng, không assume share.)
  • Thêm metadata vào YAML: recorded_against: { app, env, url, date } để debug.
  • Cảnh báo nếu folder đã có file cùng tên: hỏi overwrite hay rename.

src/scaffold.ts

  • Mở rộng ScaffoldOpts: thêm envs: string[]; apps map sang { name, urls: Record<env, string> }.
  • CLI option --apps support 2 format:
    • Mới: "rsa=ci:https://ci-rsa.frt.vn,uat:https://uat-rsa.frt.vn"
    • Backward-compat: "rsa=https://x.frt.vn" = chỉ env ci, các env khác null.
  • Thêm option --envs ci,uat,prod (default ci,uat,prod theo quyết định Q1).
  • Render template environments.yaml.tpl với schema v2.
  • Tạo folder structure: flows/{app}/{env}/ cho mọi cặp app × env, kèm _shared/ per app.
  • Command mới add-env: thêm env mới (hoặc set URL cho env null) cho 1 app.
  • Command mới clone-flow --from ci --to uat <file>: copy YAML từ folder env này sang env khác làm seed. Đổi metadata recorded_against.env. KHÔNG đổi URL trong steps (user phải tự sửa hoặc re-record).

chain-template/config/environments.yaml.tpl

Cập nhật template render schema v2 với {{ENVIRONMENTS_YAML}} + {{APPS_YAML_V2}}. Default 3 env: ci/uat/prod, app pilot chỉ có CI URL, UAT/PROD null.

7. Flow YAML — không tham số hóa

Quyết định REV 2: Mỗi env có file YAML riêng. URL hardcode trong YAML giữ nguyên. Engine KHÔNG inject hay rewrite URL.

Lý do (theo user)

  • Mỗi env (CI/UAT/PROD) có thể có DOM/data/flow khác nhau (vd UAT có thêm bước OTP, PROD có data thật, CI có dummy).
  • Test record trên CI không đảm bảo chạy được trên UAT — selectors có thể khác.
  • Tham số hóa URL tự động dễ tạo cảm giác sai "test này chạy được mọi env" → nguy hiểm khi chạy lên PROD.
  • Tách file → tester rõ ràng phải verify lại trên từng env trước khi commit.

frt-test record --app rsa --env ci -n "Tao don hang"

# 3. PROD: thường smoke test riêng, record fresh

8. Recorder behavior

Khi tester chạy frt-test record --app rsa --env uat -n "X":

  1. CLI resolve URL = config.apps.rsa.urls.uat. Nếu null → fail với gợi ý add-env.
  2. Recorder mở browser tới URL đó, ghi action.
  3. Save YAML vào flows/rsa/uat/<name>.yaml:
    • Step goto giữ URL hardcode (theo Q3).
    • Subsequent navigation cũng giữ URL hardcode.
    • Cross-domain (SSO redirect): giữ URL gốc + comment # external.
  4. Thêm metadata vào YAML: 
    recorded_against:
  app: rsa
  env: uat
  url: "https://uat-rsa-web.frt.vn"
  date: "2026-06-04"
  5. Nếu file cùng tên đã tồn tại trong folder env đó: prompt user (overwrite / rename / cancel).

clone-flow command (mới)

Dành cho user muốn dùng test CI làm seed cho UAT/PROD:

$ frt-test clone-flow --app rsa --from ci --to uat tao-don-hang.yaml

✅ Cloned: flows/rsa/ci/tao-don-hang.yaml → flows/rsa/uat/tao-don-hang.yaml
   - URL trong steps: GIỮ NGUYÊN (https://ci-rsa-web.frt.vn).
     Nếu UAT URL khác, sửa thủ công hoặc dùng find-replace.
   - recorded_against.env: ci → uat
   - recorded_against.url: cập nhật theo config UAT (nếu có), hoặc null kèm cảnh báo
   ⚠️  Verify lại trên UAT trước khi commit. DOM/flow có thể khác CI.

Logic clone-flow:

  1. Read source YAML, copy sang dest folder.
  2. Update recorded_against metadata (env, url theo config).
  3. NẾU URL UAT có sẵn trong config → optional offer find-replace URL CI → URL UAT trong steps (prompt user confirm). Default KHÔNG auto-replace để giữ explicit.

9. Scaffold template + add-app + add-env

Onboard chuỗi mới (nâng cấp)

Format đầy đủ — chỉ định URL cho mỗi env có sẵn. Env không có URL set null hoặc bỏ qua:

frt-test scaffold-chain \
  --name lab \
  --display "LAB (Xét nghiệm)" \
  --envs "ci,uat,prod" \
  --apps "rsa-lab=ci:https://ci-lab-rsa.frt.vn" \
  --apps "ecom-lab=ci:https://ci-lab-ecom.frt.vn,uat:https://uat-lab-ecom.frt.vn" \
  --output ~/work/frt-test-lab \
  --remote git@git2.fptshop.com.vn:ai-test/frt-test-lab.git \
  --push

Kết quả:

  • rsa-lab: chỉ ci có URL, uat/prod = null.
  • ecom-lab: ci+uat có URL, prod = null.
  • Folder structure tạo đầy đủ: flows/rsa-lab/{ci,uat,prod}/, flows/ecom-lab/{ci,uat,prod}/.

Backward-compat: --apps "rsa-lab=https://ci-lab-rsa.frt.vn" (1 URL không kèm env prefix) → tự assign vào env ci, các env khác null. Cảnh báo deprecation.

Add app vào chuỗi đã có

frt-test add-app \
  --name portal-lab \
  --urls "ci:https://ci-lab-portal.frt.vn,uat:https://uat-lab-portal.frt.vn"
# prod không có → tự set null

Backward-compat: frt-test add-app --name portal-lab --url https://ci-lab-portal.frt.vn (1 URL) → assign vào env ci, deprecation warning.

Add env / set URL cho env (mới)

# Set URL cho env null hoặc thay URL hiện tại

# Define env mới chưa có (vd thêm 'sit' cho 1 chuỗi đặc biệt)

10. Migration cho 8 chains đã onboard

Migration logic (theo Q3 + Q4 chốt với user):

  1. Convert config/environments.yaml v1 → v2:
    • URL hiện tại assign vào env ci (giả định all 8 chuỗi đang record trên CI).
    • Env uat + prod = null → user dùng add-env sau khi có URL.
    • Thêm schema_version: 2 + environments: block (3 env tiêu chuẩn).
  2. Move file YAML hiện có:
    • flows/rsa/foo.yaml → flows/rsa/ci/foo.yaml (dùng git mv để giữ history).
    • flows/rsa/_shared/login.yaml giữ nguyên (shared cho mọi env).

  3. KHÔNG sửa nội dung file YAML. URL hardcode giữ nguyên.
  4. Backup config: environments.yaml → environments.yaml.bak.
  5. Print summary diff cho user xem.

Q7 ⏳ Per-env settings (cần user trả lời)

Mô tả lại:

File environments.yaml hiện có block settings: ở top-level:

settings:
  timeout: 30000
  headless: false
  screenshotOnFail: true

timeout: 30000 = 30s cho mọi action. PROD network thường chậm hơn CI/UAT (qua VPN, firewall, traffic thật) → có thể cần timeout: 60000.

Câu hỏi: Có muốn override settings theo env không?

# Option A — Out of scope (đề xuất)
# Giữ settings top-level, không override theo env.
# Khi nào thực sự cần (chuỗi A báo cáo PROD timeout) thì làm plan riêng sau.

# Option B — Làm luôn
environments:
  ci:
    label: "CI"
    settings:                        # ← override per env
      timeout: 30000
  prod:
    label: "Production"
    settings:
      timeout: 60000                 # ← prod chậm hơn
      headless: true                 # ← prod chỉ chạy headless
settings:                            # ← default fallback
  timeout: 30000

Khuyến nghị: Option A — out of scope. Lý do:

  • Plan này đã đủ rộng (4 command mới, schema migration, folder restructure).
  • Chưa có user complaint cụ thể về timeout PROD chậm.
  • Khi cần thì tester có thể override per-test bằng config.timeout trong YAML.
  • Làm sau dễ hơn — schema add field optional, backward-compat.
