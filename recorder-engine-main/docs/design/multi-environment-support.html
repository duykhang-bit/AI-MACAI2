Multi-Environment Support — SDD

Project: @ai-test/recorder-engine v1.0.0 · Created: 2026-06-04 · Status: Implemented

Context

Engine had single URL per app. Testers needed multi-env (CI/UAT/PROD) support with separate test files per environment, mandatory --env flag, and migration tooling for 8 existing chains.

Requirements (from plan)

  • Schema v2: apps.{app}.urls.{env} with null support for "not yet set"
  • --env always required (no default). FRT_TEST_ENV env var override for CI/CD.
  • Each env = separate YAML file at flows/{app}/{env}/name.yaml
  • Backward-compat: v0/v1 schemas still readable
  • 4 new commands: migrate-config, add-env, envs, clone-flow
  • Pharmacy pilot migration

Design Decisions

Config Schema Detection

Auto-detect by presence of apps.X.urls map OR explicit schema_version: 2. No migration required for old chains — they keep working with v1 schema.

Resolve Algorithm

resolveBaseUrl(app?, env?, configPath?)
  → v2: lookup apps[app].urls[env] → return URL or throw
  → v1: return apps[app].baseUrl (env must be undefined or "default")
  → v0: legacy environments key converted to apps

Folder Layout

flows/{app}/{env}/*.yaml    — env-specific tests
flows/{app}/_shared/*.yaml  — shared across envs (login flows)
flows/_shared/*.yaml        — shared across all apps

Migration Strategy

Opt-in via frt-test migrate-config. Uses git mv for history preservation. Creates backup. Does NOT modify YAML content — URLs stay hardcoded per user decision.

clone-flow: No URL Replacement

When cloning test from CI to UAT, URLs in steps are kept as-is. Only recorded_against metadata is updated. User must verify/fix manually. This prevents false assumption "CI test works on UAT".

  • envs command truncates long URLs to hostname for table readability
